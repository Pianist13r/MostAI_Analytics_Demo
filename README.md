# MostAI Analytics

> Сквозная аналитика онлайн-проекта: визиты → атрибуция → выручка. Apache Superset поверх ETL-конвейера на Python с пятью моделями маркетинговой атрибуции и двумя временными режимами их применения.

**Живой стенд:** [mostai-demo.duckdns.org](https://mostai-demo.duckdns.org/superset/dashboard/1/) — открывается без логина, дашборд и его внутреннее устройство доступны в режиме просмотра. Данные обезличены.

---

## О проекте

Полноценная BI-система для онлайн-сервиса: показывает, откуда пришёл пользователь, какие касания привели к оплате и сколько на самом деле зарабатывает каждая рекламная кампания. Заменяет связку «Метрика + Директ + ручные таблицы» одним дашбордом, который пересобирается автоматически и хранит исторический контекст.

**Что закрывает дашборд:**

- **Сквозная воронка** от визита до оплаты в **пяти моделях атрибуции** (last-click, linear, time-decay, first-touch, last-significant-touch) с переключением модели в один клик
- **Два временных режима** отображения конверсий: «по дате события» (операционная аналитика — оплата на день её совершения) и «по дате касания» (полноценная маркетинговая атрибуция — оплата перекидывается на день того визита, который определила выбранная модель)
- **Полный набор метрик маркетинга** на одной странице: визиты, новые пользователи, регистрации, успешные и неуспешные оплаты, сумма оплат, среднее пополнение, время визита, глубина просмотра, процент отказов, ROAS, DRR, CR регистрация→оплата, CR пользователь→оплата, стоимость визита, регистрации, оплаты, пользователя и нового пользователя
- **15+ нативных фильтров** на боковой панели слева: временной режим, тип атрибуции, гранулярность периодов, период плашек и графиков, источник, кампания (L2), контент (L3), фраза (L4), заголовок объявления, ID объявления, браузер, страна, город, девайс, язык браузера
- **Иерархический дрилл-даун по UTM** — источник → кампания (L2) → контент (L3) → фраза (L4)
- **Галереи живых креативов Яндекс.Директа** с расходом и выручкой за выбранный период, с историчностью изменений объявлений и прямым переходом в кабинет рекламной системы
- **Кросс-фильтрация** — клик по любому селектору метрики или разреза мгновенно перестраивает все 30+ чартов дашборда
- **Защита от несовместимых срезов** — данные по расходу, кликам и показам собираются только по Яндекс.Директу; при выборе источников и разрезов, где этих данных нет, дашборд автоматически отключает зависимые от них метрики
- **Контроль качества данных** — расхождения между Метрикой, Директом и MongoDB, аномалии трафика, видимость пропусков

## Ключевые технические решения

### Два временных режима и пять моделей атрибуции

Дашборд работает в двух временных режимах, и они открывают разное количество моделей атрибуции:

- **«По дате события»** — оплаты и регистрации отображаются в день своего фактического совершения. Операционный взгляд: «сколько денег пришло сегодня». В этом режиме доступны **три модели** атрибуции — last-click, first-touch и last-significant-touch: «крайние» модели, которые относят конверсию к одному конкретному визиту цепочки, поэтому их корректно класть на дату самой конверсии.
- **«По дате касания»** — каждая конверсия перекидывается на день того визита, который определила выбранная модель. Например, оплата 12 мая по last-click из визита 5 мая → в этом режиме оплата ложится на 5 мая. Полноценный маркетинговый взгляд: «откуда пришли эти деньги». В этом режиме доступны **все пять моделей** — last-click, linear, time-decay, first-touch, last-significant-touch: режим разблокирует Linear и Time-decay, которые **распределяют** конверсию по нескольким визитам и только в проекции на дату касания дают осмысленную картину по дням.

Итого 3 + 5 = **восемь представлений** одних и тех же данных, без дублирования чартов и без перезагрузки дашборда. Переключение мгновенное — через Jinja-параметризованные виртуальные датасеты.

### Кастомная классификация источников трафика

Источник визита определяется не только по UTM-параметрам. UTM покрывают рекламный платный трафик, но органика, прямые заходы, рефералы из мессенджеров, переходы из соцсетей, внутренние перенаправления и десятки других сценариев требуют комбинированной логики. В проекте источник вычисляется из набора полей: UTM-метки, домен реферера, тип трафика по Метрике, поисковая фраза, страна и язык браузера, признак внутреннего перехода (`SITE_INTERNAL_DOMAINS`), а также наличие соответствия в специальной справочной таблице.

Эта **справочная таблица** ведётся отдельно — в ней лежат правила сопоставления «исходные признаки визита → канонический источник», специфичные именно под наш бизнес: какие домены считать рефералами от партнёров, как разделить Telegram-органику и Telegram-рекламу, куда отнести брендовые поисковые запросы и т.п. ETL применяет эти правила при сборке `visits_calculated`, и на выходе поле `last_source` (и его варианты для других моделей) уже содержит понятный бизнесу канал, а не сырой UTM или referer.

Такая классификация даёт две важные вещи: единый словарь источников во всех чартах дашборда и возможность точечно достраивать правила под новые маркетинговые активности без правки кода — достаточно добавить строку в справочную таблицу.

### Пять моделей атрибуции в одном датасете

Центральная таблица `visits_calculated` пересобирается одним проходом и хранит сразу все пять моделей: **last-click**, **linear**, **time-decay**, **first-touch** и **last-significant-touch**. Каждая конверсия (регистрация, оплата) распределяется по визитам цепочки по правилам соответствующей модели. Пересборка — атомарный swap: данные пишутся в таблицу `_new`, после успешного завершения старая таблица заменяется через `ALTER TABLE RENAME` в одной транзакции. Окно неконсистентности для дашборда — нулевое. Базовые CTE атрибуции вынесены в `_build_attribution_base_ctes()` — бизнес-правило «failed = status≠approved и нет approved в течение часа» живёт в одном месте.

### SCD Type 2 для рекламных креативов

Объявления Яндекс.Директа меняются часто: переписали текст, заменили картинку, поставили на паузу. `loader_direct_ads.py` ведёт SCD Type 2 в таблице `direct_ads` — каждое изменение создаёт новую версию с `valid_from`/`valid_to`. Дашборд показывает креатив **в том виде, в котором он крутился на момент клика**, а не текущую редакцию.

### Виртуальные датасеты с Jinja-фильтрами

Селектор «Модель атрибуции» и другие селекторы режимов не дублируют чарты на каждый вариант. Вместо этого виртуальные датасеты Superset используют Jinja-параметры:

```sql
SELECT ... FROM visits_calculated
WHERE attribution_model = '{{ filter_values("attribution_model")[0] }}'
```

Один чарт — несколько режимов отображения, переключение мгновенное.

### Селекторы метрик и разрезов из таблиц с CSS-маскировкой

Селекторы метрики и разреза в дашборде — это не нативные dropdown'ы Superset. Вместо них **обычный таблично-чарт на отдельном датасете**, где каждая строка — допустимое значение. CSS прячет заголовки и границы, превращая таблицу в вертикальный ряд кнопок.

Связь с остальными чартами идёт через **dashboard cross-filter**: клик по строке передаёт выбранное значение в Jinja-параметр виртуальных SQL других чартов, и те мгновенно перерисовываются под новый выбор. Дефолтное значение метрики подсвечивается с момента загрузки — CSS-правило красит ту строку селектора, которая соответствует дефолту в Jinja-фильтре. При клике на другую строку cross-filter обновляется, и CSS перекрашивает подсветку.

Дополнительный плюс: такие селекторы — это обычные чарты дашборда, и их можно ставить **прямо рядом с той визуализацией, которой они управляют**. Пользователь сразу видит, какой селектор на что влияет, без необходимости искать глобальный фильтр на боковой панели — управление становится интуитивнее.

### Distinct count с группировкой через трёхуровневый SQL

Superset не умеет `COUNT(DISTINCT ...) OVER (PARTITION BY ...)` напрямую. Реализовано через трёхуровневую структуру в виртуальном датасете — внутренний слой делает distinct, средний группирует, внешний агрегирует.

Нужен этот паттерн в первую очередь для **спарклайнов в KPI-плашках**: рядом с большой цифрой («1,234,567 уникальных пользователей за период») рисуется маленький график динамики метрики по дням. Чтобы получить такой ряд значений, надо для каждого дня посчитать distinct по `unified_user_id` за подходящее окно — а это и есть та самая агрегация «distinct + groupby + window», которую штатные KPI-чарты Superset не умеют. Через трёхуровневый SQL спарклайны строятся одним запросом и выглядят аккуратно над/под каждой плашкой.

### Handlebars-чарты: KPI-плашки и интерактивные галереи креативов

Все нестандартные визуальные элементы дашборда рисуются через Handlebars-чарты с собственным HTML/CSS. По сути это даёт **произвольную визуализацию любой сложности** в рамках обычного Superset-чарта: верстаешь шаблон, привязываешь данные через `{{this.field}}`, навешиваешь любой CSS — и получаешь визуализацию, которой нет среди встроенных типов. Под этот подход легли два класса элементов:

**KPI-плашки** — большие циферные показатели в шапке вкладок (визиты, оплаты, ROAS и т.д.), стилизованные под фирменный дизайн, с динамическим окрашиванием по сравнению с прошлым периодом.

**Галереи креативов** — карточка каждого живого объявления Яндекс.Директа. Карточки **интерактивные:**

- внутри карточки переключаются изображение креатива и его текстовое описание (заголовок + текст объявления)
- если описание менялось во времени, видна **вся история версий** (благодаря SCD Type 2 в `direct_ads`)
- по клику открывается **прямой переход в кабинет Яндекс.Директ** на это объявление

Галереи строятся в трёх режимах — «все», «оптимизировать» (низкий ROAS), «реактивировать» (давно не работало) — и каждая карточка тянет реальные расходы и выручку за выбранный период.

### Защита от несовместимых фильтров и разрезов

Финансово-рекламные метрики (расход, ROAS, DRR, CPC, CPM, стоимость визита, регистрации, оплаты, пользователя) считаются только по визитам из Яндекс.Директа — это единственный источник в проекте, для которого собираются данные по тратам, кликам и показам. Для других каналов (Реферальная программа, Прямой переход, мессенджеры, прочие сайты) этих данных физически нет — соответствующие метрики были бы пустыми или числились бы как `NULL/0`.

Чтобы пользователь не строил бессмысленные сравнения и не видел странных провалов на графиках, дашборд автоматически **отключает несовместимые метрики**: при выборе источника или разреза без данных по расходу зависимые от них варианты в селекторе метрики становятся неактивными, а соответствующие чарты скрываются. То же для разрезов: выборки уровня L2/L3/L4 UTM имеют смысл только в контексте Яндекс.Директа — при других источниках они тоже отключаются.

### Прогрев Redis-кэша после ETL

`loader_cache_warmer.py` дёргает Superset REST API сразу после завершения пересборки и форсирует материализацию всех чартов дашборда в Redis. Когда пользователь утром открывает страницу — она рисуется из кэша за полсекунды вместо 30+ секунд на холодный запрос.

### Единая точка управления cutoff'ами

Все даты-границы определяются в одном месте:

- `REPORT_START_DATE` (`loader_config.py`) — нижняя граница отчётности
- `_TODAY_MSK_TS` (`loader_visits.py`) — верхний cutoff «до вчерашнего дня по МСК» через `NOW() AT TIME ZONE 'Europe/Moscow'`

Меняется в одном файле — пересчитываются все модули.

## ETL модули

| Модуль | Назначение |
|---|---|
| `metrika_loader.py` | Точка входа, фиксированный порядок шагов ETL |
| `loader_config.py` | env, логирование, SQLAlchemy engine, `REPORT_START_DATE` |
| `loader_metrika.py` | Logs API + Reporting API (clientID → mongoID), миграция типов |
| `loader_mongo.py` | Рефереры, платежи, пользователи из MongoDB (атомарный swap) |
| `loader_direct.py` | Reports API Директа (AD_PERFORMANCE + CRITERIA_PERFORMANCE) |
| `loader_direct_ads.py` | Ads API + AdImages API v5 → SCD Type 2 для креативов |
| `loader_visits.py` | Пересборка `visits_calculated`, 5 моделей атрибуции |
| `loader_quality.py` | Таблица `data_quality` — контроль качества и расхождений |
| `loader_cache_warmer.py` | Прогрев Redis-кэша Superset после ETL |
| `scheduler.py` | Планировщик ежедневного полного ETL |

## Требования к источникам

- **Яндекс.Метрика** — OAuth-токен с доступом к счётчику
- **Яндекс.Директ** — тот же OAuth-токен; при агентском доступе указывается `DIRECT_CLIENT_LOGIN`
- **MongoDB** — подключение к базе с коллекциями пользователей и платежей
